## P3 Issues (Minor)

### Issue #1: Python YAML Module Not Installed in Test Environment

**Priority**: P3 (Minor)
**Status**: üü¢ Open
**Component**: Integration Tests
**Affects**: Automated YAML validation test only
**Impact**: LOW (one test skipped, manual validation works)

**Description**:

Integration test script `run-integration-tests.sh` expects Python `yaml` module for automated YAML validation (Test #2), but the module is not installed in the current test environment.

**Affected Functionality**:
- Automated YAML validation test (Test #2: YAML Validity)
- One of 15 automated tests is skipped
- Manual YAML validation still works (all files manually validated)

**Workarounds**:

1. **Manual YAML Validation** (Recommended for v1.4.0)
   ```bash
   # All YAML files have been manually validated
   # No syntax errors found
   # Safe to proceed without automated test
   ```

2. **Install PyYAML** (Optional)
   ```bash
   pip3 install pyyaml
   # Then re-run: bash .claude/memory/scripts/run-integration-tests.sh
   ```

3. **Alternative YAML Validators**
   ```bash
   # Use yamllint
   brew install yamllint
   yamllint .claude/memory/**/*.yaml

   # Use online validators
   # https://www.yamllint.com/
   # https://jsonformatter.org/yaml-validator
   ```

**Root Cause**:
- PyYAML is an optional Python dependency, not guaranteed in all environments
- Test script assumes PyYAML is available

**Fix Plan**:

**v1.4.1 (Patch Release)**:
- Update test script to check if PyYAML is installed before running YAML validation
- Provide clear error message if missing: "PyYAML not installed. Install with: pip3 install pyyaml"
- Document PyYAML as optional pre-requisite in README
- Alternatively, use YAML parser built into bash (if available)

**Timeline**: Non-blocking for v1.4.0, fix in v1.4.1 (or v1.5.0)

**Testing**: 14/15 tests pass, 1/15 skipped (93% pass rate, acceptable)

**Risk Assessment**: MINIMAL (manual validation complete, no YAML errors exist)

### Issue #2: Emergency Rollback SOP Not a Standalone Document

**Priority**: P3 (Minor)
**Status**: üü¢ Open
**Component**: Standard Operating Procedures
**Affects**: Emergency rollback documentation only
**Impact**: LOW (procedure covered in deployment SOP)

**Description**:

Emergency rollback procedure is embedded in `deployment.md` SOP (Stage 6: Rollback) but is not a standalone document for quick reference during emergencies.

**Affected Functionality**:
- Emergency responders may need to search through deployment SOP to find rollback steps
- No dedicated "emergency_rollback.md" for quick access

**Workarounds**:

1. **Use Deployment SOP Section** (Current Workaround)
   ```bash
   # Open deployment SOP
   cat .claude/memory/sop-templates/deployment.md | grep -A 50 "Stage 6: Rollback"

   # Or view full deployment SOP
   # Section: "Stage 6: Rollback (If Needed)"
   ```

2. **Create Bookmark** (Quick Reference)
   ```bash
   # Create quick reference alias
   alias rollback-help='cat .claude/memory/sop-templates/deployment.md | grep -A 50 "Stage 6: Rollback"'
   ```

3. **Extract Section Manually**
   ```bash
   # Extract rollback section to temporary file (if needed)
   sed -n '/Stage 6: Rollback/,/Stage 7:/p' .claude/memory/sop-templates/deployment.md > /tmp/emergency-rollback.md
   ```

**Current Rollback Procedure** (from deployment.md):

```markdown
## Stage 6: Rollback (If Needed)

### When to Rollback
- Health check failures
- Critical errors in production
- User-facing functionality broken
- Data corruption detected
- Security breach

### Rollback Steps

1. **Stop traffic to new version**
   ```bash
   # Switch load balancer to old version
   # Or use blue-green deployment switch
   ```

2. **Revert to previous version**
   ```bash
   git checkout <previous-tag>
   # Or use deployment rollback command
   ```

3. **Verify rollback successful**
   ```bash
   # Run health checks
   # Verify user functionality
   ```

4. **Communicate status**
   - Notify stakeholders
   - Update status page
   - Create incident report

5. **Root cause analysis**
   - Identify failure cause
   - Document in postmortem
   - Plan fix for next release
```

**Root Cause**:
- Initial SOP design embedded rollback in deployment workflow
- Did not anticipate need for standalone emergency reference

**Fix Plan**:

**v1.4.1 or v1.5.0**:
- Create standalone `emergency_rollback.md` SOP
- Include:
  - Quick reference checklist
  - One-page summary
  - Links to detailed deployment SOP
  - Common failure scenarios
  - Contact information
  - Escalation paths

**Timeline**: Non-blocking for v1.4.0, create in v1.4.1 or v1.5.0

**Risk Assessment**: MINIMAL (deployment SOP fully covers rollback procedure)

## Hotfix Policy

### When to Release Hotfix

Hotfix releases (v1.4.0.1, v1.4.0.2, etc.) are created for:
- **P0 (Blocker)** issues only
- Data loss or corruption
- Security vulnerabilities (CVE)
- Production outages

### Hotfix Process

1. **Identify P0 issue**
2. **Create hotfix branch**: `git checkout -b hotfix/v1.4.0.1 v1.4.0`
3. **Fix issue** with minimal changes
4. **Test thoroughly** (regression tests)
5. **Merge and tag**: `git tag v1.4.0.1`
6. **Release immediately**
7. **Backport to main branch**
